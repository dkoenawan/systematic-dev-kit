# Feature Spec: User Management

> Generated by /systematic-dev-kit:plan | Date: 2026-02-17 | Status: Draft
> Relates to: New standalone feature

## Summary

Administrators can manage user accounts across the system — create new users, assign roles, search the user directory, and disable accounts. Regular users can view and edit their own profile. This centralizes identity management that was previously handled through direct database access.

## Complexity Profile

| Signal | Applies? | Notes |

## Database Layer

### New Models

```prisma
enum UserStatus {
  ACTIVE
  INACTIVE
  SUSPENDED
}

model User {
  id        String     @id @default(cuid())
  email     String     @unique
  name      String
  password  String     // bcrypt hash
  status    UserStatus @default(ACTIVE)
  lastLogin DateTime?
  createdAt DateTime   @default(now())
  updatedAt DateTime   @updatedAt

  roles     UserRole[]
  profile   UserProfile?

  @@map("users")
}

model Role {
  id          String   @id @default(cuid())
  name        String   @unique   // ADMIN, MANAGER, USER
  description String?
  createdAt   DateTime @default(now())

  users       UserRole[]

  @@map("roles")
}

model UserRole {
  userId    String
  roleId    String
  assignedAt DateTime @default(now())

  user      User @relation(fields: [userId], references: [id], onDelete: Cascade)
  role      Role @relation(fields: [roleId], references: [id], onDelete: Cascade)

  @@id([userId, roleId])
  @@map("user_roles")
}

model UserProfile {
  id        String  @id @default(cuid())
  userId    String  @unique
  phone     String?
  bio       String?
  timezone  String  @default("UTC")

  user      User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("user_profiles")
}
```

### Data Rules / Constraints

| Rule | Implementation |
|------|---------------|
| Email must be unique | `@unique` decorator on `email` field |
| User status is enumerated | Prisma enum: `ACTIVE`, `INACTIVE`, `SUSPENDED` |
| Password is never returned in API responses | Excluded in query select / response mapping |
| Deleting a user cascades to roles and profile | `onDelete: Cascade` on relations |
| Role names are unique | `@unique` decorator on `name` field |

## Backend Layer

### Commands

| Name | Endpoint | Method | Auth | Description |
|------|----------|--------|------|-------------|
| CreateUserCommand | POST /api/users | POST | ADMIN | Create a new user with initial role assignment |
| UpdateUserCommand | PUT /api/users/:id | PUT | ADMIN or SELF | Update user details (admin can update any, user can update own profile) |
| DeleteUserCommand | DELETE /api/users/:id | DELETE | ADMIN | Soft-delete by setting status to INACTIVE |
| AssignRoleCommand | POST /api/users/:id/roles | POST | ADMIN | Assign a role to a user |
| RemoveRoleCommand | DELETE /api/users/:id/roles/:roleId | DELETE | ADMIN | Remove a role from a user |
| UpdateProfileCommand | PUT /api/users/:id/profile | PUT | SELF | Update own profile (phone, bio, timezone) |

### Queries

| Name | Endpoint | Method | Auth | Description |
|------|----------|--------|------|-------------|
| GetUserQuery | GET /api/users/:id | GET | ADMIN or SELF | Get user details with roles and profile |
| ListUsersQuery | GET /api/users | GET | ADMIN | List users with pagination, search by name/email, filter by role/status |
| GetProfileQuery | GET /api/users/me | GET | AUTH | Get current user's own profile |
| ListRolesQuery | GET /api/roles | GET | ADMIN | List all available roles |

  name?: string;
  status?: "ACTIVE" | "INACTIVE" | "SUSPENDED";

interface UpdateProfileRequest {
  phone?: string;
  bio?: string;
  timezone?: string;

// List Users
interface ListUsersRequest {
  page?: number;       // default: 1

  sortOrder?: "asc" | "desc";               // default: desc

interface ListUsersResponse {

  total: number;

  limit: number;

// Role Assignment
interface AssignRoleRequest {

### Validation Rules

| Field | Rule | Error Message |
|-------|------|---------------|
| email | Required, valid email format, max 255 chars | "Valid email address is required" |
| name | Required, 1-100 characters | "Name must be between 1 and 100 characters" |
| password | Required on create, min 8 chars, 1 uppercase, 1 number | "Password must be at least 8 characters with 1 uppercase letter and 1 number" |
| roleIds | Required on create, at least 1 valid role ID | "At least one role must be assigned" |
| phone | Optional, E.164 format if provided | "Phone must be in valid format" |
| bio | Optional, max 500 characters | "Bio must be under 500 characters" |
| timezone | Valid IANA timezone string | "Invalid timezone" |
| limit | 1-100 | "Limit must be between 1 and 100" |

### Error Scenarios

| Scenario | Status | Message | Handling |
|----------|--------|---------|----------|
| Email already exists | 409 | "A user with this email already exists" | Show inline error on email field |
| User not found | 404 | "User not found" | Redirect to user list with toast |
| Insufficient permissions | 403 | "You don't have permission to perform this action" | Show error toast, keep on current page |
| Cannot delete own account | 400 | "You cannot deactivate your own account" | Show error toast |
| Invalid role ID | 400 | "Role not found" | Show error toast |
| Last admin removal | 400 | "Cannot remove the last admin role from the system" | Show error toast with explanation |

### Frontend/Backend Boundary

**Balanced approach.** Server handles all data validation, authorization checks, password hashing, and pagination. Frontend handles form state, client-side validation (for instant feedback), and UI state like sort/filter preferences. Search is server-side (database LIKE/full-text) since user data can grow unbounded.

### UI Patterns Used

| Pattern | Where | Notes |
|---------|-------|-------|
| Data table | UserListPage | Sortable by name, email, created date; filterable by role and status; paginated |
| Form with validation | UserCreatePage, UserEditPage, ProfilePage | React Hook Form with Zod schema; inline field errors |
| Modal / Dialog | UserDetailPage → RoleAssignmentDialog | For adding/removing roles without leaving the detail page |

### State Management

| State | Source | Strategy | Rationale |
|-------|--------|----------|-----------|
| User list | GET /api/users | Cache & revalidate | List is read-heavy; stale-while-revalidate keeps UI responsive while ensuring freshness |
| Single user | GET /api/users/:id | Cache & revalidate | Same rationale; cache key by user ID |
| Current profile | GET /api/users/me | Cache & revalidate | Rarely changes; cache with long stale time |
| Form state | Local | React Hook Form | Standard form library; no need for global state |
| Search/filter params | URL search params | URL state | Shareable, bookmarkable, survives refresh |
| Role list | GET /api/roles | Cache & revalidate | Roles rarely change; cache with long stale time |

### Navigation

**Entry points**:
- Sidebar navigation link "Users" (visible to ADMIN role only)
- Profile icon in top navigation → "My Profile" (all authenticated users)

**Exit points**:
- After creating user → redirect to user detail page
- After editing user → redirect to user detail page
- Back button on detail/edit pages → return to user list
- Profile page is standalone (no redirect after save, show success toast)
